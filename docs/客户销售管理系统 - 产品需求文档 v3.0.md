
## 一、产品概述

### 1.1 产品愿景与目标

#### 1.1.1 产品愿景

打造一个高效、协同的客户全生命周期管理系统，支持销售、客服、管理三个角色无缝协作，从潜在客户获取到成为IB合作伙伴的全流程管理。

#### 1.1.2 核心目标

- **提升转化率**：通过精准的任务分配和跟进机制，提升潜客到注册、注册到入金的转化率
- **保障客户体验**：分配规则确保客户问题有明确责任人，避免多人重复联系
- **优化运营效率**：自动化规则、任务池、工单系统减少人工协调成本
- **数据驱动决策**：实时绩效分析帮助管理层快速识别问题和机会

### 1.2 核心业务场景

#### 1.2.1 销售场景

- 从公海池中获取潜在客户
- 通过多渠道（电话、WhatsApp、Email）联系客户
- 引导客户完成注册、KYC、入金、首笔交易
- 为符合条件的客户创建IB协议

#### 1.2.2 客服场景

- 接收来自CRM同步的客户工单
- 通过沟通中心处理实时对话（Live Chat、WhatsApp等）
- 主动创建工单记录客户问题
- 升级复杂问题给管理层或转交给销售

#### 1.2.3 管理场景

- 管理销售和客服团队成员
- 分配任务和设置自动化规则
- 审批IB协议
- 监控团队绩效和客户生命周期状态

### 1.3 系统角色

系统有4种角色：

| 角色 | 职能 |
|---|---|
| Sales | 负责拉新，维护客户关系 |
| Support | 与客户沟通，答疑解惑，解决客户遇到的问题 |
| Manager | 管理Sales，Support团队 |
| Admin | 系统配置，账户分配 |

#### 2. 分配规则

| 分配方式 | 触发条件 | 分配逻辑 | 说明 |
|---|---|---|---|
| 手动分配 | Manager创建任务 | 选择特定Sales或Support | 精准控制 |
| 自动分配 | 自动化规则触发 | 根据规则匹配（语言/地区/负载） | 提升效率 |
| 任务池 | 任务未分配或被退回 | Sales/Support主动认领 | 灵活调配 |

#### 3. 状态流转

```
Pending（待处理） → In Progress（进行中） → Completed（已完成）
                                          → Failed（失败）
                                          → No Response（客户无响应）
```

| 状态值 | 中文名称 | 英文名称 | 说明 | 下一步操作 |
|---|---|---|---|---|
| `pending` | 待处理 | Pending | 已分配但未开始 | 开始处理 |
| `in_progress` | 进行中 | In Progress | 正在执行中 | 完成、标记失败、标记无响应 |
| `completed` | 已完成 | Completed | 成功完成 | 归档 |
| `failed` | 失败 | Failed | 任务失败 | 重新分配或关闭 |
| `no_response` | 无响应 | No Response | 客户无响应 | 由自动化规则决定是否重试 |

**No Response重试机制**：
- 通过自动化规则配置重试策略（重试间隔、最大重试次数N）
- 系统记录每次重试次数
- 达到最大重试次数N后，任务自动标记为 `failed`，不再重试

### 2.4 IB Application

#### 1. 来源

IB协议的创建遵循以下标准链路：

```
Step 1: 客户在CRM提交IB申请
Step 2: CRM生成Ticket并推送事件 ticket.created（category=ib_application）到本系统
Step 3: 本系统接收并创建工单，自动分配给Support
Step 4: Support核对后转交给Sales（Escalate → Forward to Sales）
Step 5: Sales查看客户填写的IB申请并联系客户沟通细节
Step 6: Sales创建IB Deal并提交Manager审批（后续审批流程见本节“审核”）
```

| 来源 | 说明 |
|---|---|
| Sales创建 | Sales基于Support转交的IB申请Ticket创建IB Deal，提交Manager审批 |
| Manager创建 | Manager直接为IB客户创建IB Deal |

#### 2. 表单字段

| 字段名 | 类型 | 必填 | 验证规则 | 说明 |
|---|---|---|---|---|
| 返佣比例 | 数字输入 | 是 | > 0 | 例如：$8.00 |
| 返佣类型 | 下拉选择 | 是 | - | 每手(per_lot) / 百分比(percentage) / 固定金额(fixed) |
| 月度目标 | 文本输入 | 是 | - | 例如："100手"或"$50,000" |
| 生效日期 | 日期选择 | 是 | >= 今天 | 协议开始日期 |
| 到期日期 | 日期选择 | 否 | > 生效日期 | 协议结束日期 |
| 额外支付 | 数字输入 | 否 | >= 0 | IB市场开发费用 |
| 支付频率 | 下拉选择 | 条件必填 | 有额外支付时必填 | 一次性(one_time) / 月度(monthly) / 季度(quarterly) |
| 备注 | 多行文本 | 否 | - | 额外说明 |

#### 3. 审核

**状态流转**：

```
Pending Approval（待审批） → Approved（已批准） → Active（生效中） → Expired（已过期）
                          → Rejected（已拒绝） → 修改后重新提交
```

| 状态值 | 中文 | 英文 | 说明 |
|---|---|---|---|
| `pending_approval` | 待审批 | Pending Approval | Sales提交后等待Manager审批 |
| `approved` | 已批准 | Approved | Manager审批通过，通知Sales，Manager通知Operation在CRM端审核（人工处理，非系统自动同步） |
| `rejected` | 已拒绝 | Rejected | Manager拒绝，Sales可修改后重新提交 |
| `active` | 生效中 | Active | 到达生效日期后自动变为生效中 |
| `expired` | 已过期 | Expired | 超过到期日期后自动变为已过期 |

**审批操作**（仅Manager）：

| 操作 | 必填字段 | 后续动作 |
|---|---|---|
| 批准 | 无 | Deal状态→approved，通知Sales，Manager通知Operation在CRM端审核 |
| 拒绝 | 拒绝原因（必填）、修改建议（选填） | Deal状态→rejected，通知Sales |

**审批历史记录**：

每次操作记录在审批历史中，包含：操作时间、操作类型（created/submitted/approved/rejected/modified）、操作人、原因、状态变化。

**状态自动流转**：系统通过定时扫描（如每日一次）检查所有IB Deal，到达生效日期的 `approved` 自动转为 `active`，超过到期日期的 `active` 自动转为 `expired`。当状态流转为 `expired` 时，系统发布内部事件 `ib.deal_expired`。

### 2.7 Automation

#### 1. 属性

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| 规则名称 | 文本 | 是 | 便于识别的规则名称 |
| 启用状态 | Toggle开关 | 是 | 是否生效 |
| 触发方式 | 单选 | 是 | 事件驱动 / 定时扫描 / 两者结合 |
| 触发条件 | 条件组 | 是 | 见触发条件定义 |
| 执行动作 | 动作组 | 是 | 见执行动作定义 |
| 创建人 | 系统自动 | - | 谁创建的规则 |
| 创建时间 | 系统自动 | - | 规则创建时间 |
| 最后修改 | 系统自动 | - | 最后修改时间 |

#### 2. 触发方式

| 方式 | 说明 | 适用场景 |
|---|---|---|
| 事件驱动 | 收到外部事件时立即检查匹配的规则并执行 | 客户状态变更、入金、KYC通过等实时事件 |
| 定时扫描 | 按设定的时间间隔定时扫描所有客户/任务是否匹配条件 | 客户休眠检测、No Response重试等时间类条件 |

两种方式可结合使用，同一规则可同时支持事件触发和定时扫描。

#### 3. 触发条件

触发条件支持“条件组 + 逻辑运算符（AND/OR）”组合：
- 组内条件可配置为 AND 或 OR
- 多个条件组之间可配置为 AND 或 OR
- 支持表达式示例：`(任务优先级 = High OR Urgent) AND (任务状态 = Pending)`

**客户条件**：

| 条件类型 | 字段名 | 输入方式 | 选项 |
|---|---|---|---|
| 客户状态 | clientStatus | 多选下拉 | Lead / Registered / KYC Pending / KYC Rejected / KYC Approved / Funded / Active / Dormant |
| 客户国家 | clientCountry | 多选下拉 | 国家列表 |
| 客户语言 | clientLanguage | 多选下拉 | 语言列表 |

**任务条件**：

| 条件类型 | 字段名 | 输入方式 | 选项 |
|---|---|---|---|
| 任务类型 | taskType | 多选下拉 | 见Task任务类型枚举 |
| 任务优先级 | taskPriority | 多选下拉 | Low / Medium / High / Urgent |
| 任务状态 | taskStatus | 多选下拉 | Pending / In Progress / Completed / Failed / No Response |
| 重试次数 | retryCount | 数字比较 | < N（小于最大重试次数） |

**人员/分组条件**：

| 条件类型 | 字段名 | 输入方式 | 选项 |
|---|---|---|---|
| 负责人分组 | assigneeGroup | 多选下拉 | 见2.6 Group列表 |

**时间条件**：

| 条件类型 | 字段名 | 输入方式 | 说明 |
|---|---|---|---|
| 最后活跃距今 | lastActivityDays | 数字（天） | 客户最后活跃时间距今超过N天 |
| 重试间隔 | retryInterval | 数字（天） | 上次No Response后等待X天再重试 |

**事件条件**（支持外部事件 + 内部系统事件）：

| 条件类型 | 字段名 | 输入方式 | 选项 |
|---|---|---|---|
| 外部事件 | externalEvent | 多选下拉 | 见4.8 外部系统推送事件 |
| 内部系统事件 | internalEvent | 多选下拉 | 见4.8.1 内部系统事件 |

#### 4. 执行动作

| 动作类型 | 说明 |
|---|---|
| 分配给特定人员 | 指定具体的Sales或Support |
| 分配给Group | 指定目标Group，系统按配置策略分配（round_robin / least_load / skill_match） |
| 按语言匹配分配 | 匹配语言能力相符的人员 |
| 按区域匹配分配 | 匹配负责区域相符的人员 |
| 创建任务 | 自动创建指定类型的任务 |
| 更新客户状态 | 将客户状态更新为指定状态（如Dormant） |
| 通知Manager | 发送通知给Manager |
| 升级给Manager | 自动升级 |

**Group分配策略**：

| 策略 | 英文标识 | 分配逻辑 | 适用场景 |
|---|---|---|---|
| 轮询分配 | `round_robin` | 按组内成员固定顺序依次分配，不考虑当前负载 | 标准化任务、追求平均分配 |
| 最小负载分配 | `least_load` | 优先分配给当前未完成任务数最少的成员 | 控制积压、均衡工作量 |
| 技能匹配分配 | `skill_match` | 先按语言/区域/专长标签筛选可处理成员，再在候选中按最小负载分配 | KYC/IB等专业任务、跨语言任务 |

**默认策略与回退顺序**：
- 默认优先级：`skill_match` > `least_load` > `round_robin`
- 若`skill_match`无匹配成员，自动回退到`least_load`
- 若`least_load`候选为空（如成员均离线或不可接单），回退到`round_robin`

#### 5. 规则示例

**示例1：中国KYC通过客户自动分配**
```
触发方式：事件驱动
触发条件：客户状态 = KYC Approved AND 国家 = 中国 AND 语言 = 中文
执行动作：按语言匹配分配（中文） + 按区域匹配分配（中国） + 创建任务（入金引导） + 通知Manager
```

**示例2：高优先级任务通知Manager**
```
触发方式：事件驱动
触发条件：任务优先级 = High 或 Urgent
执行动作：通知Manager
```

**示例3：客户休眠检测**
```
触发方式：定时扫描
触发条件：最后活跃距今 > N天
执行动作：客户状态→Dormant + 创建任务（重新激活） + 通知Manager
```

**示例4：No Response重试**
```
触发方式：定时扫描
触发条件：任务状态 = No Response AND 重试次数 < N AND 重试间隔 >= X天
执行动作：创建新任务（同类型） + 重试次数+1
```

**示例5：IB到期自动续签跟进**
```
触发方式：事件驱动
触发条件：内部系统事件 = ib.deal_expired
执行动作：创建任务（IB续签跟进） + 通知Manager
```

**示例6：IB申请工单分组路由**
```
触发方式：事件驱动
触发条件：外部事件 = ticket.created AND 工单分类 = ib_application
执行动作：分配给Group（IB Support Group） + 通知Manager
```

## 三、业务流程

### 3.1 Sales工作流

该流程从获取潜在客户开始，经过注册、KYC认证、审核、入金等多个环节，最终完成客户转化。在这个过程中，Sales角色承担着关键节点的推进工作，同时涉及到任务创建、分配与协助的协作机制。需要注意的是，Sales与客户的联系方式为个人号（如个人微信、个人手机号、公司派发的邮箱，个人WhatsApp），而非公司统一配发的官方账号。

#### 3.1.1 客户转化

将潜在客户转化为系统客户，并协助客户注册、KYC认证、审核、入金，最终完成客户转化。

```
Lead(潜在客户) → Register → Client(registered) → Submit KYC → Client(kyc_pending) → Approve → Client(kyc_approved) → Fund → Client(funded)
```

**客户获取方式**

Sales获取潜在客户有两种途径：
1. **主动认领**：从 Public Pool（潜在客户池）中浏览潜在客户，点击"Claim"认领到自己名下
2. **Manager分配**：Manager在 Public Pool 中选择潜在客户，指定分配给某位Sales

认领/分配后，该客户从 Public Pool 移入 Sales 的 My Clients 列表，Sales成为该客户的负责人。

**各阶段详细说明**

| 阶段 | 客户状态 | 外部事件 | 系统动作 | Sales行为 |
|---|---|---|---|---|
| 潜在客户获取 | lead | — | 客户出现在 Public Pool | 从 Public Pool 认领客户，或由 Manager 分配 |
| 客户注册 | registered | `client.registered` | 更新客户状态；触发自动化规则（如创建"引导KYC"任务） | 联系客户，引导完成注册；协助解决注册过程中的问题 |
| KYC提交 | kyc_pending | `client.kyc_submitted` | 更新客户状态 | 确认客户已提交KYC；告知客户审核周期 |
| KYC审核通过 | kyc_approved | `client.kyc_approved` | 更新客户状态；触发自动化规则（如创建"引导入金"任务） | 通知客户审核通过；引导客户进行首次入金 |
| KYC审核拒绝 | kyc_rejected | `client.kyc_rejected` | 更新客户状态；触发自动化规则（如创建"协助重新提交KYC"任务） | 联系客户告知拒绝原因；协助客户修正资料并重新提交 |
| 客户入金 | funded | `client.funded` | 更新客户状态；触发自动化规则 | 确认入金成功；开始引导客户交易 |

**客户退回机制**

Sales可以将已认领的客户退回 Public Pool：
- 操作：在 My Clients 中选择客户，点击"Release"
- 效果：客户回到 Public Pool，Sales不再是该客户的负责人
- 适用场景：客户长期无响应、客户区域/语言不匹配等

#### 3.1.2 任务

Sales角色通过任务系统进行日常工作管理，任务来源包括系统自动创建（由自动化规则触发）和Manager手动创建。

**任务日常工作流**

```
Sales上线
  ↓
查看 My Tasks（按优先级排序）
  ↓
选择任务 → 查看任务详情（客户信息、任务要求、历史记录）
  ↓
执行任务：
  - 联系客户（个人号：微信/手机/邮箱/WhatsApp）
  - 记录沟通结果（Add Call / Send SMS / Send Email）
  ↓
判断结果：
  ├─ 客户已响应 → 完成任务（Complete），记录结果
  ├─ 客户未响应 → 标记 No Response，等待自动化规则触发重试
  └─ 无法处理 → 退回任务池（Return to Pool）
  ↓
处理下一个任务
```

**任务来源与创建**

| 来源 | 创建方式 | 示例 |
|---|---|---|
| 自动化规则 | 外部事件触发自动创建 | 客户注册后自动创建"引导KYC"任务 |
| Manager手动 | Manager在任务管理中创建并分配 | 指派Sales跟进特定客户 |
| 自动化规则（定时） | 定时扫描条件触发 | 客户3天未活动，自动创建"跟进"任务 |

**任务分配机制**

- Manager创建任务后可直接分配给指定Sales，或放入Task Pool
- 自动化规则创建的任务：优先分配给客户的负责Sales，若无负责人则进入Task Pool
- Sales可从Task Pool主动认领（Claim）任务

**No Response处理**

- Sales标记任务为"No Response"后，自动化规则根据配置自动创建重试任务
- 重试间隔和最大次数由自动化规则配置（如：间隔2天，最多3次）
- 达到最大重试次数后，任务标记为"failed"，不再自动创建

#### 3.1.3 Ticket

Sales处理的Ticket主要来自Support转交（Escalate → Forward to Sales）的工单。

**Ticket处理流程**

```
收到Ticket通知
  ↓
查看 My Tickets
  ↓
打开Ticket详情（客户信息、问题描述、历史工单）
  ↓
处理Ticket：
  ├─ 直接回复解决
  ├─ 需要沟通 → 通过个人号联系客户 → 回复Ticket并记录
  └─ 超出权限 → 升级（Escalate）给Manager
  ↓
关闭Ticket（Close）
```

- **Ticket归属**：CRM同步工单先分配给Support；涉及销售职责的工单由Support转交给Sales
- **沟通记录**：Sales处理Ticket时，相关的沟通记录应手动添加到Ticket或关联的客户记录中
- **升级机制**：超出Sales权限范围的问题，可升级给Manager处理

#### 3.1.4 IB协议

Sales协助客户完成IB（Introducing Broker）协议签订。

**标准流程（客户通过CRM申请）**

```
客户在CRM提交IB申请
  ↓
CRM生成IB申请Ticket → 推送 ticket.created（category=ib_application）事件
  ↓
本系统创建工单并自动分配给Support
  ↓
Support核对后转交给Sales（Escalate → Forward to Sales）
  ↓
Sales查看客户填写的IB申请信息 → 联系客户沟通细节 → 点击"Create IB Deal"
  ↓
填写IB Deal（佣金方案、月度目标等） → 提交Manager审批
  ├─ 审批通过 → 状态变为 approved（待生效）→ 到达生效日期后自动变为 active
  │            → Manager通知Operation在CRM端审核 → CRM推送 client.ib_activated（写入IB Code/等级）
  └─ 审批拒绝 → 通知Sales → Sales联系客户说明原因
```

**客户直接联系Sales的场景**

```
客户向Sales表达IB意向
  ↓
Sales提示客户：请先在CRM客户端提交IB申请
  ↓
等待CRM生成并同步IB申请Ticket（`ticket.created`）→ 回到标准流程
```

- **前置条件**：存在由CRM同步的IB申请工单（`ticket.created`且`category=ib_application`），并已由Support转交给Sales
- **IB身份写入时机**：Manager审批通过并通知Operation在CRM端审核后，由`client.ib_activated`事件回写IB Code、IB等级等信息
- **Sales职责**：跟进Support转交的IB申请工单、与客户沟通细节、创建IB Deal、跟进审批进度、通知客户结果
- **到期处理**：IB Deal到期后状态变为 `expired`，Sales可协助客户续签（创建新的IB Application）

### 3.2 Support工作流

Support团队通过公司官方渠道接收并处理客户请求，核心工作包括：实时对话处理、Ticket跟��、任务执行。与Sales不同，Support使用公司统一配发的官方渠道（Business WhatsApp、Livechat、Hotline、Support Email），客户主动联系Support，而非Support主动联系客户。

#### 3.2.1 沟通渠道与实时对话

Support的核心工作是通过沟通中心（Communication Center）实时处理来自各渠道的客户咨询。

**支持的沟通渠道**

| 渠道类型 | 具体渠道 | 说明 |
|---|---|---|
| 在线聊天 | Livechat | 网站/App内的在线聊天工具，实时沟通 |
| 即时通讯 | Business WhatsApp | 企业版WhatsApp，客户通过即时消息沟通 |
| 邮件 | Support/Services Email | 客服支持邮箱，客户通过邮件发送问题 |
| 电话 | Hotline Phone number | 客服热线电话，客户通过语音通话咨询问题 |

**日常工作流**

```
Support上线
  ↓
进入沟通中心（Communication Center）
  ↓
查看各渠道待回复对话（按未读/等待时间排序）
  ├─ Live Chat：实时网页聊天
  ├─ WhatsApp：Business WhatsApp消息
  ├─ Email：客服邮箱收件
  └─ Phone：热线电话来电
  ↓
选择对话 → 查看客户上下文（右侧面板：客户信息、余额、最近工单）
  ↓
处理对话：
  ├─ 直接回复解决
  ├─ 需要后续跟踪 → 点击"Create Ticket"创建工单（source: internal_created）
  └─ 超出权限 → 创建Ticket并升级（Escalate）
  ↓
继续处理下一个对话
```

**与Sales沟通方式的关键区别**

| 维度 | Sales | Support |
|---|---|---|
| 沟通渠道 | 个人号（个人微信、手机、邮箱、WhatsApp） | 公司官方渠道（Business WhatsApp、Livechat、Hotline、Support Email） |
| 沟通方向 | Sales主动联系客户 | 客户主动联系Support |
| 记录方式 | 手动在系统中添加沟通记录 | 沟通中心自动记录对话 |
| 工作界面 | 任务详情页 → 联系客户 | 沟通中心 → 接收对话 |

#### 3.2.2 Ticket处理

Support的Ticket来自两个途径：CRM同步和内部创建。

**Ticket来源**

| 来源 | source值 | 说明 |
|---|---|---|
| CRM同步 | `crm_issue` | 客户在CRM客户端提交的Issue（含IB申请场景），通过`ticket.created`事件同步 |
| 内部创建 | `internal_created` | Support在沟通中心处理对话时，点击"Create Ticket"创建 |

**Ticket处理流程**

```
收到Ticket（CRM同步或自己创建）
  ↓
查看 My Tickets（按优先级和创建时间排序）
  ↓
打开Ticket详情（客户信息、问题描述、历史工单、关联对话）
  ↓
处理Ticket：
  ├─ 直接回复解决 → 标记 Resolved → 关闭
  ├─ 需要沟通 → 通过沟通中心联系客户 → 回复并更新Ticket
  ├─ 超出权限 → 升级给Manager（Escalate → Submit to Manager）
  └─ 属于Sales职责 → 转交给Sales（Escalate → Forward to Sales）
  ↓
关闭Ticket（Close）
```

- **Ticket归属**：通过分配规则决定由哪位Support负责
- **沟通中心联动**：Support在沟通中心处理实时对话时，可点击"Create Ticket"将对话转为工单跟踪
- **IB申请工单处理**：当`category=ib_application`时，Support核对基础信息后应转交给Sales继续处理

#### 3.2.3 任务

Support角色也通过任务系统进行工作管理，任务由Manager手动创建或自动化规则触发。

**任务日常工作流**

```
查看 My Tasks（按优先级排序）
  ↓
选择任务 → 查看任务详情
  ↓
执行任务（与Ticket/沟通中心联动）
  ↓
判断结果：
  ├─ 任务完成 → Complete，记录结果
  ├─ 无法处理 → 退回任务池（Return to Pool）
  └─ 需要协助 → 联系Manager
  ↓
处理下一个任务
```

**任务分配机制**

- Manager创建任务后可直接分配给指定Support，或放入Task Pool
- 自动化规则创建的任务进入Task Pool，由Support主动认领（Claim）
- Support可随时将任务退回Task Pool

#### 3.2.4 升级机制

Support遇到超出权限或能力范围的问题时，可通过升级机制转交处理。

**升级流程**

```
Support判断需要升级
  ↓
点击"Escalate"按钮
  ↓
填写升级表单：
  - 升级类型：Submit to Manager / Forward to Sales
  - 优先级：Low / Medium / High / Urgent
  - 升级原因（必填）
  - 备注（选填）
  ↓
提交 → Ticket状态变为 on_hold
  ↓
Manager收到通知 → 处理：
  ├─ 直接处理并返回Support
  ├─ 转交给Sales
  └─ 拒绝升级，返回Support重新处理
```

**升级场景**

- 客户问题涉及销售承诺或合同条款 → Forward to Sales
- 客户投诉需要管理层介入 → Submit to Manager
- 技术问题需要更高权限操作 → Submit to Manager

### 3.3 Manager工作流

Manager负责团队管理、任务分配、审批决策和绩效监控，是Sales Team和Support Team的管理者。Manager拥有全局数据视图和最高操作权限。

#### 3.3.1 团队管理

Manager负责管理Sales Team和Support Team的成员。

```
Manager进入Team管理
  ↓
查看团队成员列表（姓名、角色、状态、语言、区域、任务负荷）
  ↓
操作：
  ├─ 添加成员：填写信息（角色、语言、区域、最大任务容量等）
  ├─ 编辑成员：修改属性（语言、区域、容量、专长等）
  ├─ 切换状态：Active / Busy / Offline
  └─ 管理分组：创建/编辑Group，给成员打标签
```

**状态切换影响**

| 状态变更 | 系统行为 |
|---|---|
| Active → Busy | 停止向该成员分配新任务 |
| Active → Offline | 停止分配新任务；提醒该成员流转未完成任务（支持一键流转或退回任务池） |
| Busy/Offline → Active | 恢复正常任务分配 |

#### 3.3.2 任务管理

Manager创建任务并分配给Sales或Support执行。

```
Manager创建任务
  ↓
填写任务信息（客户、任务类型、优先级、描述）
  ↓
分配方式：
  ├─ 直接分配：指定某位Sales/Support
  └─ 放入Task Pool：等待成员认领
  ↓
跟踪任务进度（查看All Tasks视图）
  ↓
必要时重新分配任务
```

#### 3.3.3 IB协议审批

Manager审批Sales提交的IB Application。

```
收到IB Application审批通知
  ↓
查看Pending Approval列表
  ↓
打开申请详情（客户信息、佣金方案、Monthly Target等）
  ↓
审批决策：
  ├─ 批准（Approve）→ 通知Sales → Manager通知Operation在CRM端审核
  └─ 拒绝（Reject）→ 填写拒绝原因 → 通知Sales
```

#### 3.3.4 Ticket升级处理

Manager处理Support升级上来的Ticket。

```
收到升级通知
  ↓
查看升级的Ticket详情（问题描述、升级原因、客户信息）
  ↓
处理决策：
  ├─ 直接处理 → 解决后返回Support关闭
  ├─ 转交给Sales → Sales接手处理

```

#### 3.3.5 客户管理

Manager拥有全局客户视图和完整操作权限。

- 查看所有客户（Clients视图），不受负责人限制
- 编辑客户信息（包括状态修改）

#### 3.3.6 自动化规则管理

Manager创建和管理自动化规则，实现任务自动分配和通知。

```
Manager进入Automation管理
  ↓
查看规则列表（名称、状态、条件、动作）
  ↓
操作：
  ├─ 创建规则：定义触发条件 + 执行动作
  ├─ 编辑规则：修改条件或动作
  ├─ 启用/禁用规则：开关切换
  └─ 查看执行日志
```

#### 3.3.7 绩效监控

Manager通过Performance模块监控团队表现。

- **Sales团队指标**：新增客户数、注册转化率、入金转化率、总入金金额、任务完成率
- **Support团队指标**：总工单数、平均处理时长、首次响应时间、工单解决率、升级率
- **个人排行榜**：按角色分别展示成员绩效排名
- **客户生命周期分析**：各阶段分布、漏斗转化率、流失分析

### 3.4 自动化引擎工作流

描述自动化规则引擎如何运行，包括事件驱动和定时扫描两种触发方式的完整处理链路。

#### 3.4.1 事件驱动流程

```
外部事件到达（如 client.registered）
  ↓
事件分发器接收
  ↓
匹配所有已启用的自动化规则（按条件逐一评估）
  ↓
命中规则 → 执行动作：
  ├─ 创建任务（自动分配给客户负责人，或进入Task Pool）
  ├─ 发送通知（通知Manager/Sales/Support）
  ├─ 更新客户状态
  └─ 升级给Manager
  ↓
记录执行日志（规则名、触发事件、执行动作、执行时间）
```

#### 3.4.2 定时扫描流程

```
定时器触发（如每小时一次）
  ↓
扫描所有已启用的、含时间类条件的规则
  ↓
逐条评估：查询满足条件的数据（如"3天未活动的客户"）
  ↓
命中 → 执行动作（同上）
  ↓
记录执行日志
```

#### 3.4.3 No Response自动重试

```
Sales标记任务为 No Response
  ↓
自动化规则监听内部事件 task.no_response
  ↓
检查条件：重试次数 < 最大次数N
  ├─ 满足 → 等待配置的间隔时间 → 创建新的重试任务（重试次数+1）
  └─ 不满足（已达上限）→ 标记任务为 failed，不再重试
```

### 3.5 外部事件消费流程

描述系统如何接收和处理来自CRM/MT5等外部系统的事件。

#### 3.5.1 正常消费流程

```
外部系统（CRM/MT5/其他）发布事件
  ↓
RabbitMQ 接收并路由到对应队列
  ↓
本系统消费者从队列拉取事件
  ↓
事件处理：
  ├─ 解析事件数据（校验格式、提取字段）
  ├─ 更新本地数据（Client状态、余额、交易记录等）
  ├─ 触发自动化引擎（将事件传递给规则匹配器）
  └─ 记录事件日志
  ↓
确认消费（ACK）
```

#### 3.5.2 异常处理

```
事件消费失败
  ├─ 数据格式错误 → 记录错误日志，丢弃消息（Dead Letter Queue）
  ├─ 处理逻辑异常 → 重试（最多N次）→ 仍失败则进入Dead Letter Queue
  └─ 系统不可用 → 消息留在队列，恢复后继续消费
```

### 3.6 客户状态流转

描述客户状态如何由外部事件驱动自动流转，以及每次流转触发的系统动作。

**状态流转图**

```
lead → [client.registered] → registered
                                  ↓
                          [client.kyc_submitted] → kyc_pending
                                                      ↓
                                          ┌─ [client.kyc_approved] → kyc_approved
                                          │                              ↓
                                          │                      [client.funded] → funded
                                          │                                          ↓
                                          │                                  [client.traded] → active
                                          │                                                      ↓
                                          │                                          [Automation: 最后活跃超N天] → dormant
                                          │                                                      ↑                    │
                                          │                                                      └─ [client.traded] ──┘
                                          └─ [client.kyc_rejected] → kyc_rejected
                                                                        ↓
                                                              [client.kyc_submitted] → kyc_pending（重新提交）
```

**每次状态流转的系统动作**

1. 更新本地Client记录（状态、时间戳、关联数据）
2. 触发自动化引擎评估相关规则
3. 可能自动创建任务（如"引导KYC"、"引导入金"）
4. 可能发送通知（通知客户负责人状态变更）

### 3.7 通知机制

### 4.8 外部系统推送事件（External Push Events）

**事件消费架构**：
- 外部系统（CRM、MT5等）将事件推送到 RabbitMQ 消息队列
- 本系统启动时订阅对应队列，异步消费事件
- 每种事件对应独立的队列，便于独立扩展和故障隔离

```
外部系统（CRM / MT5 / 其他） → RabbitMQ → 本系统（消费者）
```

**事件列表**：

| 事件名称 | 英文标识 | 来源系统 | 触发时机 | 推送数据 | 影响模块 |
|---|---|---|---|---|---|
| 客户注册 | `client.registered` | CRM | 客户在CRM完成注册 | 基础信息、注册时间 | Client, Task |
| KYC提交 | `client.kyc_submitted` | CRM | 客户提交KYC资料 | KYC状态、提交时间 | Client |
| KYC审核通过 | `client.kyc_approved` | CRM | CRM审核通过KYC | KYC通过时间 | Client, Task |
| KYC审核拒绝 | `client.kyc_rejected` | CRM | CRM审核拒绝KYC | 拒绝原因、拒绝时间 | Client, Task |
| 客户入金 | `client.funded` | CRM | 客户完成入金 | 入金金额、入金时间、账户余额 | Client, Task |
| 客户出金 | `client.withdrawal` | CRM | 客户发起出金申请 | 出金金额、出金状态、申请时间 | Task |
| 余额变动（Wallet） | `client.wallet_updated` | CRM | CRM Wallet余额变化 | 当前余额、变动金额、变动原因 | Client, Task |
| 余额变动（MT5） | `client.mt5_balance_updated` | MT5 | MT5 Account余额变化 | 当前余额、变动金额、变动原因 | Client, Task |
| 客户交易 | `client.traded` | MT5 | 客户完成交易 | 交易时间、交易信息 | Client |
| 入金金额汇总 | `client.deposit_summary` | CRM | 定期推送客户入金汇总数据 | 客户ID、入金总额、入金笔数、时间范围 | Performance |
| 状态变更 | `client.status_changed` | CRM | 客户状态在CRM中变更 | 新状态、旧状态、变更时间 | Client |
| IB资格激活 | `client.ib_activated` | CRM | Manager审批通过后，Operation在CRM端完成IB资格审核 | IB Code、IB类型、IB等级 | Client, IB Application |

- `ticket.created` 事件：CRM当前工单没有分类（category）字段，需要CRM做好适配，在推送数据中包含工单分类信息（见2.3 Ticket工单分类枚举），并支持`ib_application`分类。

### 4.11 数据脱敏规则（Data Masking）

**各角色脱敏策略**

| 角色 | 客户联系方式 | 说明 |
|---|---|---|
| Sales | 完整显示（自己负责的客户） | — |
| Support | 脱敏显示 | 可点击"查看完整联系方式"申请查看，记录审计日志 |
| Manager | 完整显示 | — |
| Admin | 不涉及 | Admin不访问客户模块 |

**脱敏规则**

| 数据类型 | 原始数据 | 脱敏后显示 |
|---|---|---|
| 手机号码 | 13812345678 | 138\*\*\*\*5678 |
| 邮箱地址 | john.doe@example.com | j\*\*\*\*\*e@example.com |
| 客户姓名 | 不脱敏 | 不脱敏 |

### 4.12 审计日志（Audit Log）

以下敏感操作需记录审计日志，日志内容包括：操作人、操作时间、操作对象、操作详情。

| 操作类型 | 触发场景 | 记录内容 |
|---|---|---|
| 查看完整联系方式 | Support点击"查看完整联系方式" | 操作人、客户ID、查看时间 |
| 客户数据变更 | Manager编辑客户信息 | 操作人、客户ID、变更字段、旧值、新值、时间 |
| 用户状态变更 | Manager切换成员状态 | 操作人、目标用户、旧状态、新状态、时间 |
| 角色权限变更 | Admin编辑自定义角色 | 操作人、角色名、变更内容、时间 |
| Ticket升级 | Support/Manager发起升级 | 操作人、Ticket ID、升级类型、升级原因、时间 |
| IB审批操作 | Manager审批IB Application | 操作人、申请ID、审批结果、审批备注、时间 |
| 系统设置变更 | Admin修改系统配置 | 操作人、配置项、旧值、新值、时间 |
| 自动化规则变更 | Manager创建/编辑/启停规则 | 操作人、规则ID、操作类型、时间 |
